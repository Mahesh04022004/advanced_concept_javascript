Q. Explain the difference between undefined, null, and a variable that has not been declared. How would you safely check for each?

undefined means a variable exists but has not been assigned a value. null is an intentional assignment that represents the absence of a value. An undeclared variable does not exist in the current scope at all and accessing it directly throws a ReferenceError.

To check for undefined, use value === undefined. To check for null, use value === null. To safely test whether a variable exists without risking a ReferenceError, use typeof value === "undefined".

One important JavaScript quirk is that typeof null returns "object" due to a historical bug. Another common trap is that null == undefined is true, while null === undefined is false. Using strict equality is generally preferred unless intentionally checking for both null and undefined together.

Q. Can you explain the difference between == and === in terms of the Abstract Equality and Strict Equality algorithms? Give examples where == produces surprising results.

=== uses the Strict Equality algorithm. It compares both type and value and performs no type coercion. If the operands have different types, it immediately returns false.

== uses the Abstract Equality algorithm. When the operand types differ, JavaScript applies a set of coercion rules to convert values before comparing them. This can lead to surprising results such as false == 0, "" == 0, [] == false, and null == undefined all evaluating to true.

Because implicit coercion can make code difficult to reason about, most production code prefers ===. One notable exception is value == null, which is a concise way to check for both null and undefined.

What is the Temporal Dead Zone (TDZ) in let and const? How does it differ from var hoisting?

let and const declarations are hoisted, but unlike var, they are not initialized during the memory creation phase. From the start of their scope until the declaration is evaluated, they remain in a Temporal Dead Zone (TDZ). Any attempt to access them during this period throws a ReferenceError.

In contrast, var is hoisted and initialized to undefined, so accessing it before its declaration does not throw an error and simply returns undefined.

The TDZ was introduced to catch bugs early and prevent accidental use of variables before initialization, making code more predictable and safer.

Q. Explain the internal [[Environment]] slot and how closures preserve the lexical scope chain even after outer function returns.

Every JavaScript function object contains an internal hidden slot called [[Environment]]. When the function is created, this slot stores a reference to the lexical environment in which the function was defined. When the function later executes, variable lookup starts in its local scope and then follows the environment chain referenced by [[Environment]].

A closure occurs when a function survives beyond the lifetime of its outer function. Even after the outer function returns, its lexical environment is preserved because the inner function's [[Environment]] still references it. As long as the closure remains reachable, the outer variables cannot be garbage collected. This is why closed-over variables continue to exist and retain state across multiple function calls.

Q. What happens when you attempt to access a property on null or undefined? What does optional chaining (?.) evaluate to in those cases?

Accessing a property on null or undefined throws a TypeError because neither value can act as an object reference. For example, null.name and undefined.name both result in runtime errors.

Optional chaining (?.) changes this behavior by performing a nullish check before property access. If the left-hand side evaluates to null or undefined, evaluation short-circuits and the expression returns undefined instead of throwing.

For example, user?.name is roughly equivalent to user == null ? undefined : user.name. Optional chaining can be applied repeatedly, such as user?.address?.city, to safely traverse deeply nested object structures without explicit null checks.

Q. How does the new operator work internally? Write a polyfill for it.

Internally, the new operator performs four main steps:

Creates a new empty object.
Sets the object's prototype to the constructor's prototype.
Invokes the constructor with this bound to the new object.
Returns the created object unless the constructor explicitly returns another object, in which case that object is returned instead.

A polyfill can be implemented by creating an object with Object.create(Constructor.prototype), invoking the constructor using apply, and then returning either the constructor's returned object or the created instance.
